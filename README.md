# ChromaQuran

A free **black-canvas Quran video studio** — part of the *Wisdom From Quran* project. Pick a surah + verse
range, a reciter and a typeface, preview it on a pure-black 9:16 canvas synced to the recitation, then render a
downloadable **1080×1920 MP4**. Built with Next.js 16 (App Router) + React 19 + TypeScript.

## What's real
- **Quran text:** all 114 surahs (Uthmani Arabic + Sahih International English), bundled as static JSON in
  `public/data/surahs/` (generated by `scripts/fetch-quran.mjs`). Loaded on demand, served by nginx — no runtime API.
- **Recitation:** real per-ayah audio from [everyayah.com](https://everyayah.com) (13 reciters). Playback follows the
  real audio; the verse advances when each ayah ends.
- **Export:** a real render pipeline — Playwright rasterizes the composition at 1080×1920 (one frame per ayah) and
  `ffmpeg` muxes the frames with the per-ayah audio into an H.264/AAC MP4. Runs server-side via an in-process job
  queue with progress.

## Local development
```bash
npm install
npx playwright install chromium   # one-time: render browser
npm run dev                        # http://localhost:3000
```
`ffmpeg` ships via `ffmpeg-static` (no system install needed).

To regenerate the Quran data (rarely needed):
```bash
node scripts/fetch-quran.mjs
```

## Production (self-hosted, behind nginx)
```bash
npm ci
npx playwright install --with-deps chromium   # --with-deps installs the Linux libs Chromium needs
npm run build
npm run start                                  # listens on :3000 (set PORT to change)
```
Run it under a process manager (systemd / PM2 / Docker) so it stays up. Then point nginx at it:

```nginx
server {
  server_name chromaquran.rayanemelzi.dev;
  client_max_body_size 50m;            # rendered MP4s

  location / {
    proxy_pass http://127.0.0.1:3000;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_read_timeout 600s;           # renders can take a while
  }
}
```

### Render notes
- Rendered MP4s are written to `renders/` (gitignored) and streamed via `/api/render/[id]/file`.
- Concurrency is limited so renders don't overwhelm the box. Tune with `CQ_RENDER_CONCURRENCY` (default `1`).
- The worker runs **in the Next server process**. For higher scale, move `lib/render` to a separate worker + a
  Redis/BullMQ queue (the API and UI don't change).
- Add a cron/job to prune old files in `renders/`.
